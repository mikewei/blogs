## 概述

如果只把 `claw-code` 看成一个热门开源项目，容易把重点看偏。它真正值得研究的地方，不在于“它是否复刻了 Claude Code”，而在于它给了我们一个很少见的观察窗口：**一个成熟的 Agentic Coding 系统，底层到底是怎样组织起来的。**

这几年行业里已经出现了不少面向编程的 AI 工具。有些更像“会写代码的聊天框”，有些更像“带自动化的编辑器插件”，还有一些则开始演化成真正的 agent runtime。它们表面上都在做代码生成、修改、搜索、执行，但从系统架构看，其实不是一类东西。

我看完 `claw-code` 的整体实现后，最大的判断是：**Claude Code 这类系统的核心竞争力，很可能不在单次模型调用，而在一整套围绕上下文、工具、权限、会话和环境编排出来的运行时。** 换句话说，模型只是引擎，真正决定产品上限的，是那台“车”本身。

而 `claw-code` 的价值，正是在于它把这台车的骨架露出来了。

## 先说结论：Agentic Coding不是功能堆叠，而是运行时设计

很多人看 coding agent，容易从功能列表切入：能不能读文件、能不能改代码、能不能跑命令、能不能联网。这个角度没错，但不够深。因为这些能力单独拿出来并不稀奇，真正难的是把它们组织成一个稳定、可恢复、可控、可扩展的系统。

一个成熟的 Agentic Coding 产品，至少要同时解决下面几类问题：

- 它如何理解当前项目，而不是只理解当前 prompt
- 它如何决定什么时候说话，什么时候调用工具
- 它如何在多轮执行中维持状态，而不是每轮重新开始
- 它如何限制自身行为，避免一步走错把工作区搞坏
- 它如何接入外部系统，而不是永远只活在本地终端里

从这个视角看，`claw-code` 让人看到的不是一个“会写代码的聊天机器人”，而是一个更接近操作系统或 runtime 的东西。它管理的对象不是一段回答，而是一次次连续的开发动作。

这和一些偏“问答式”的编程助手有本质差异。后者的中心是回答质量，前者的中心是**任务闭环**。回答只是过程，闭环才是产品。

## claw-code让我们看清的第一件事：真正复杂的是Harness，而不是模型接口

`claw-code` 当前保留的上游表面规模相当大，TS-like 文件超过 `1900`，顶层目录 `35` 个，模块重心大量落在通用工具、组件、服务、hooks 这类位置，而不是单一“模型调用”模块上。这个分布其实非常说明问题。

它说明一个成熟 agent 系统的大量复杂度，来自这些不那么性感但极其关键的部分：

- 上下文如何收集
- 工具如何定义与筛选
- 权限如何建模
- 会话如何存储与恢复
- 输出如何被转成可继续执行的下一步
- 远程环境、代理、企业接入如何兼容

这也是为什么我一直觉得，行业里很多对 Agentic AI 的讨论仍然偏表面。大家喜欢讨论模型排名、基准分数、上下文窗口，但对真正决定可用性的 harness 设计，讨论远远不够。

如果把模型比作 CPU，那么 harness 更像操作系统。CPU 再强，没有进程调度、权限隔离、文件系统、设备驱动，最后也只是一个裸芯片。Claude Code 这类产品真正厉害的地方，很可能就在这层“操作系统化”的工程能力。

## 第二件事：这类系统的核心不是单轮对话，而是可持续推进的任务循环

从 `claw-code` 的结构里，很容易看出一个鲜明特征：它不是围绕“消息”组织的，而是围绕“回合”组织的。

这听上去像措辞区别，但实际上是架构分水岭。

传统聊天式产品的基本单位是“一问一答”。而 coding agent 的基本单位更像：

```text
理解任务
  -> 生成下一步动作
  -> 调工具
  -> 接收结果
  -> 更新状态
  -> 再判断下一步
```

也就是说，它不是简单回答用户，而是在持续推进一个任务图。

这正是为什么这类系统天然会长出一些“聊天产品不太需要”的能力：

- turn loop
- transcript
- usage tracking
- session resume
- compaction
- tool result replay

这些能力单独看都不神奇，但它们一起出现时，背后代表的是一种产品哲学：**系统默认自己会长期工作，而不是回答完这一句就结束。**

这一点在行业内其实是一个很重要的分野。

像一些编辑器内 AI 助手，更偏向“在 IDE 工作流中插入智能补全与局部改写”；而像 `aider`、`OpenHands`、`Codex CLI`、Claude Code 这一类，则明显更接近“任务代理”。它们不是附着在编辑器里的一个功能，而是在争夺“开发流程主循环”的控制权。

`claw-code` 之所以值得看，正是因为它把这种主循环意识表现得很明显。

## 第三件事：上下文管理，比大模型本身更像产品分水岭

很多人会把“上下文”理解成把更多代码塞进 prompt 里，但这只是非常初级的一层。真正有经验的系统设计者会知道，所谓上下文管理，至少包含三件事：

- 找到什么该被放进来
- 决定什么不该被放进来
- 在多轮运行后仍保持上下文可控

`claw-code` 折射出的一个重要事实是，Claude Code 这类系统并不是把 prompt 当成一段固定模板，而是把它当成一种运行时产物。目录、日期、项目指令、配置、git 状态、本地记忆，这些东西都会参与“系统提示词”的形成。

这背后的含义很深。

今天很多团队做 agent，仍然停留在“写一个很长的 system prompt”阶段。但真正进入产品化之后，你会发现 prompt 不再是一篇文案，而更像一个编译结果。它来自很多来源，需要分层、去重、裁剪、注入边界，还要考虑何时刷新、何时压缩。

从这个角度说，Agentic Coding 的一个核心竞争点不是“谁的 prompt 写得更漂亮”，而是**谁更像在做 prompt compiler**。

这也是我认为很多外界观察低估 Claude Code 的地方。大家看到的是回答风格，看不到的是它对项目状态的组织能力。而后者往往更难，也更不容易被简单复制。

## 第四件事：权限系统不是安全附属品，而是Agent系统的基础设施

如果一个 agent 只能读，不能写，问题很简单；如果它既能写文件，又能跑 shell，还能联网、调用外部服务，问题就完全变了。

这时候权限系统不再只是“加个确认弹窗”这么简单，而必须成为架构一级概念。

`claw-code` 透露出来的权限模型很典型，也很合理：只读、工作区可写、完全访问，逐级提升。更关键的是，权限不是对“用户”一次性授权，而是对“工具调用”逐次判断。这种建模方式非常重要，因为 agent 世界里的风险不是登录风险，而是**动作风险**。

这点和传统 SaaS 的权限设计不一样。SaaS 更多是“你能不能看到这个页面”；agent 则是“此时此刻，这个动作该不该发生”。一个是静态授权，一个是动态裁决。

很多团队做 agent 时会低估这一点，觉得先把能力做出来，权限以后再补。但经验上看，后补的权限往往会把整个系统打碎，因为它会反过来影响工具协议、交互流程、错误处理和用户心智模型。

所以从一个架构师角度讲，谁把权限作为中心设计，谁才真正理解 agent 产品。谁把权限当边角料，谁大概率还停留在 demo 阶段。

## 第五件事：MCP、Remote、Proxy这些“边角能力”，其实暴露了产品野心

一个只服务个人开发者的本地工具，不一定需要考虑太多 remote、proxy、OAuth、外部服务编排。但一旦产品想走向团队、企业或更复杂的工作流，这些东西就会从“高级功能”变成“主干能力”。

`claw-code` 很有意思的一点是，它把这些能力放得很靠内核，而不是作为外围插件看待。这说明它所映射的原系统，从设计上就不是一个纯本地助手，而是一套准备接入更大环境的 agent runtime。

这与行业里另外两类产品形成对比。

第一类是“编辑器增强型”产品，它们往往更重视交互和局部体验，但在跨环境执行、远程接入、统一工具协议上不一定有那么重的设计。

第二类是“脚本自动化型”产品，它们擅长命令驱动、任务串联，但在多租户、团队接入、权限边界和远程治理上，通常没有那么完整。

Claude Code 这类系统明显更接近第三类：它想成为一套通用 agent runtime。`claw-code` 让这种野心看得很清楚。

## 但也要泼一点冷水：claw-code能揭示骨架，未必能揭示灵魂

如果只讲优点，这篇文章就没什么意思了。站在一个比较冷静的角度看，`claw-code` 也有明显边界，而且这些边界非常值得讲清。

首先，它现在最强的是“结构镜像能力”，不是“行为复现能力”。这意味着它很适合帮助我们看清系统分层、模块边界、概念设计，但并不能天然证明原系统最关键的体验细节已经被理解了。

而在 agent 产品里，很多真正决定体验的东西，恰恰不在静态结构里，而在动态策略里，例如：

- 何时触发工具，何时继续思考
- 多工具之间如何排序
- 失败后何时重试，何时回退
- 什么时候压缩上下文，压缩掉什么
- 什么时候选择保守，什么时候选择激进

这些策略往往才是“产品灵魂”。它们像调度器、像启发式系统、像无数业务经验叠加出来的隐性规则。结构能映射出来，经验未必能映射出来。

其次，`claw-code` 当前双轨结构也说明它自己仍处在过渡状态。Python 这条线偏快，适合铺开表面；Rust 这条线偏稳，适合收敛 runtime。这个方向是对的，但也有代价：**它一边像研究仓库，一边又想像产品仓库，容易让读者对它的成熟度产生错觉。**

说得直白一点，作为“理解 Claude Code 架构的样本”，它很有价值；但作为“可以直接拿来判断最终产品能力”的依据，它还远远不够。

## 这件事对行业的启发是什么

我觉得 `claw-code` 带来的最大启发，不是告诉大家“可以复刻某个产品”，而是提醒大家：Agentic Coding 的竞争已经开始从模型层，转向运行时层。

未来几年，真正拉开差距的，很可能不是谁先接入了哪个最强模型，而是下面这些能力谁做得更系统：

- 项目上下文编译
- 工具协议统一
- 会话压缩与恢复
- 细粒度权限和审批
- 远程执行与企业接入
- 可观测性、成本控制与失败恢复

这其实和云计算、数据库、中间件的发展很像。早期大家比拼的是“能不能跑起来”，后来比拼的是“能不能稳定地跑、长期地跑、在复杂环境里跑”。Agent 现在也开始走到这个阶段了。

所以如果今天你在做内部 coding agent，或者在看某个外部产品值不值得押注，我会建议少看一点 demo 视频，多看一点它的 runtime 设计。尤其要看：

- 它的工具边界是否清楚
- 它的状态管理是否严肃
- 它的权限模型是否可扩展
- 它对项目上下文的处理是否体系化

这些东西通常不如“自动写了一个网页”那样吸引眼球，但它们才决定系统能不能从玩具变成基础设施。

## 结语

`claw-code` 不是一面完美镜子，但它是一面很有价值的镜子。它让我们看到，像 Claude Code 这样的系统，本质上不是“一个很会写代码的模型”，而是一套把模型、工具、记忆、权限和环境组织成工作闭环的运行时。

从架构视角看，这类系统最难的部分，从来不是把某个 API 接进来，而是把不确定的模型能力，约束进一个长期可用、可控、可恢复、可扩展的工程系统。

这也是我读完 `claw-code` 后最想强调的一点：**Agentic AI 的下一场竞争，拼的不会只是智能本身，而是智能被安放在什么样的系统里。**
