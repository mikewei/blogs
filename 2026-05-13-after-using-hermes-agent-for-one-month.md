从一个多月前Hermes-Agent开始出圈，我开始关注它，到最近我已经把主要工作流从 OpenClaw 全部迁移到了 Hermes。

真正让我切换的，不仅是Harness能力本身，而是 Hermes 第一次让我感觉：Agent 不只是一个会调用 tools 的聊天机器人，而是一个有确定性的灵活可控的个人计算运行环境。

下面来聊一聊具体的一些使用感受。

---

## 一、先说说亮点

用 Hermes 的第一感觉就是**干净**。能明显感觉是有顶层设计的严谨工程，而不是 vibe coding 垒出来的实验项目。CLI 清晰，配置简洁，设计挺符合工程师的使用习惯。高频使用的功能都有命令行，简单可靠。

基本上所有的功能开箱即用，不需要一开始就折腾一堆设置。遇到问题的时候，直接问 LLM，即使是一些bug，Agent自己查代码大都能轻松解决。上手体验和使用体感这块比 OpenClaw 还是清晰易用了不少。

### 高效 Context Window 管理

Hermes 给我最大的感受之一，是它开始把 **context window** 当作一种真正**稀缺的系统资源**来设计。使用中经常能感受到它对有效利用 context window 的追求，并且围绕这个目标也配套了一系列完善的功能界面。

首先，通过 **Profile** 可以把任务按类型和场景划分大类，不同类别的上下文天然隔离，且可以减少无关依赖。在会话层面，**session 的命名、恢复等操作接口**设计得很到位，使用中能精确地管理每段上下文的生命周期。配上**紧凑的多级 memory 管理**——强调记住重要的事情，但不会无节制地堆砌 history——短期记忆、会话搜索、持久记忆各司其职，互不污染。

还有一个贴心细节：状态栏里有 **context window 使用率提示**，还有多少余量一目了然，方便判断要不要压缩或开新会话。

### Profile 灵活分角色

Hermes 的 Profile，简单说就是**一套独立的 Agent 配置和状态**。每个 profile 有自己独立的 soul、配置、记忆和技能，互不干扰。

我欣赏它的地方不只是"能划分"，而是**几乎零负担**。一键克隆一个独立 profile，结合 alias 命令，你可以根据使用场景轻松创建很多 profile 而没有任何心理负担。比如，我现在常用的有 writer（擅长写作、呈现）、analyst（擅长分析表格、懂价值投资）、reminder 等等——好比给不同场景准备了不同的工具箱，进哪个门拿哪套工具。

### 安全易用的 Docker Backend

一开始我会以为Docker模式重且交互不方便，但实际用下来觉得挺好用，Hermes已将Docker backend打磨得挺好了，特别有几个好处：

- **通过 volumes 清晰定义 IO 边界**：任务的输入目录、输出目录、整个运行环境都能精确控制。在沙箱里跑，健壮性明显好很多
- **默认配置就挺省心**：实际用下来发现比想象中问题少得多，一些便利性问题通过persistent env、tools call等设计已经解决了
- 目前我大部分任务都已经默认用 Docker 来跑了，可靠且简单

### Tools & Skills

Hermes 内置了大量的高质量的工具，覆盖了日常干活中的常见场景。也不用太担心工具太多占用过多上下文资源，因为不同的 profile 可以精确配置启用哪些工具——writer 不用装投资工具，analyst 不用装邮箱工具，各取所需。

另外Hermes也有很好的扩展性，你可以方便地安装/编写skills，添加MCP，通过Plugins增加tools以及CLI commands。

### 记忆与自省

Hermes 的记忆系统设计比较紧凑，包括短期记忆、搜索历史会话，也能接入外部 memory 插件来扩展记忆能力。

我观察它的记忆更新还是比较严谨的，不会乱记一些东西，只有符合特定要求或用户显式提示才会更新短期记忆。

还有它著名的自省机制，它会自己定期 review 和反思任务执行情况，通过 `skill_manage` 随时 patch skill 的内容。完成一个复杂任务之后，能自动沉淀成 skill，下次遇到类似场景直接复用，不用重新摸索。这个过程让它在使用中具备了一定的**自我进化能力**。

而且这套机制也是相对透明的，你可以随时手工 review 它的 memory 和 skill 内容，做一些优化和调整。

### 速度快

虽然是Python实现，但执行速度明显比OpenClaw快很多（同样的LLM Provider下比较），这也促使我将一些工具更快地切换过来。我猜可能还是实现质量更高一些吧。

## 二、也发现了一些小缺陷

### 文档超前于实现

这个挺奇葩的：不少用户手册上已经写了的参数或者功能，实际并没有完整实现。比如配置模板里写了 `docker_env` 参数，但不生效。再比如 CLI command plugin 机制文档写得很完整了，代码却没跟上。

我不禁怀疑——难道是先写用户手册再来开发的？

### PR 治理节奏慢

项目里有些明显的 bug 已经有 PR 了，但合并速度比较慢，特别是 P2、P3 级别的。可能维护者在优先处理高优先级的需求和新功能开发，中低优先级的 fix 就被搁置了。

问题倒不至于影响日常使用，但看着 issue 列表里躺了几个月的已知问题，总是有点遗憾。

### 审美不太在线

实话实说，Hermes 在起名字、Logo、WebUI 等方面的设计审美……不太敢恭维。不过好在这也不算是大问题。

## 三、我的工具沉淀

一个 Agent 系统是否真正好用，不仅取决于Harness能力，还看它是否容易嵌入已有 workflow，变成自己的“运行时生态”。而 Hermes 在这方面感觉有很强的可塑性。

我也尝试绕它补齐了一些让自己更顺手的基础工具：

- [pomfret](https://github.com/mikewei/pomfret)：一个类OpenRouter的轻量网关，方便在多agent部署时快速配置切换路由，同时也可以方便地跟踪LLM请求和做Prompt分析。
- [whiteboard-mcp](https://github.com/mikewei/whiteboard-mcp)：一个白板MCP服务，可以在任何终端打开白板，并让 Agent 把内容投到白板上阅读，比在终端里滚动舒服很多。
- [hermes-jobctl](https://github.com/mikewei/hermes-jobctl)：一个类 kubectl 风格的定时任务管理工具，可以用声明式描述来管理和编排各种定时作业。

这些工具都是从实际使用的痛点中总结而来的，不一定适合所有人，但对我自己的 workflow 帮助挺大。

---

## 结语

总体下，我觉得Hermes更适合对技术有一点理解，喜欢确定性，想更精确地操控Agent的用户。我也简单对比过 Manus 等商业付费 Agent 软件，感受是：Hermes 通过调优可以达到 90% 以上付费软件的效果，而使用成本要低得多，而且是完全的白盒可控。

Agent 可能正逐渐成为新的个人计算运行环境，我也将继续基于Hermes-Agent优化我的工作流，持续探索 Agent 的能力边界。
